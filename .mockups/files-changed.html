个人推荐

从「最少破坏现状 + 最快交付完整新体验」的角度，推荐 方向 A · 顶部双 Tab 作为 v1。理由：

  • 现有 FindingsWorkspace 与 FindingDetailPanel 都不必动，老用户工作流零回归
  • Files changed 是一个用户熟悉的、可独立交付的 destination，前端实现可拆解为：FileTree + FileDiffPane + InlineFindingOverlay 三件套
  • off-diff finding 不需要额外结构（仍在 Findings tab 里）
  • SSE 增量、prep 早期可用、Files changed → Findings 之间的跳转都自然

后续可演进：发现用户主要在 Files changed Tab 上完成工作后，再考虑把 Findings Tab 内容简化为「off-diff + summary」、把 Files changed 变默认（即过渡到方向 C 的形态）。

还需要你决策的几个点（无论选哪个方向都要回答）
  1. inline finding 展示形态： (a) 完整卡片直接内嵌（GitHub 风，占据 diff 流空间）； (b) 仅 gutter 上的彩色 dot/数字标记 + 点击右侧弹出 detail； (c) 折叠摘要 + 点击展开。
  2. diff 视图模式：仅 unified？还是 unified/split 双模（react-diff-view 已支持，但 split 需要更多宽度）。
  3. 文件状态机：是否引入 GitHub 风格的「Viewed ✓」勾选并持久化（per session）？还是只标记 finding 已读 / 已忽略？
  4. 大文件 / 二进制 / lockfile：策略：默认折叠（点击展开） / 直接隐藏 / 给出「在外部打开」按钮。生成 / lockfile 类应该有默认排除规则吗？
  5. finding 与代码行的锚定：findings.json 已有 file/line/startLine；但 agent 可能产出指向已删除行或上下文行的 finding。落点策略：严格匹配新增行 / 容忍上下文行 / fallback 到文件顶部？现有 diff-line-check 已做了部分校验，需要复用还是放宽？
  6. SSE 时序：用户已经滚到某个文件正在阅读，agent 新产出的 finding 命中其他文件，应：(a) 静默追加 (b) 顶部 toast 通知 (c) 文件树上文件名加 badge 闪烁。
  7. 是否同时引入「行级评论」能力：用户能否在 diff 上手动新增一条 finding（点击 + 号）？这扩大了 review 工具定位，但与「agent 主导」的现状有冲突。
